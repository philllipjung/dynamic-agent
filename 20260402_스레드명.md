# 스레드명 변경 기능 구현 완료 보고서

## 📋 개요
**날짜**: 2026-03-27
**기능**: 메서드 실행 중 스레드명을 TraceId/SpanId 기반으로 변경
**상태**: ✅ 구현 완료 및 테스트 성공
**작업 시간**: 약 5시간 (문제 해결 포함)

## 🎯 목표

메서드가 실행될 때 해당 요청의 TraceId와 SpanId를 포함한 고유한 스레드명으로 변경하여 eBPF/커널 레벨 트레이싱에서 추적 가능하게 만듦

**스레드명 형식:**
```
{TraceId 7자}{SpanId 6자}{이니셜}{숫자}
예: 26cd7480d92c2DBG6176
   -------^^^^^^^^^-^^^----
   Trace   Span    이니숫자
   7자     6자     섬 4자
```

## 📝 소스 변경사항

### 1. ThreadNameHelper 클래스 생성 (신규)

**파일**: `bytebuddy-advice/src/main/java/com/javaagent/bytebuddy/helper/ThreadNameHelper.java`

**전체 코드:**
```java
package com.javaagent.bytebuddy.helper;

import io.opentelemetry.api.trace.Span;
import io.opentelemetry.api.trace.SpanContext;
import io.opentelemetry.context.Context;

import java.util.concurrent.ThreadLocalRandom;
import java.util.regex.Pattern;

/**
 * 스레드명 관리 Helper
 *
 * OpenTelemetry TraceId와 SpanId를 사용하여 고유한 스레드명 생성
 */
public class ThreadNameHelper {

    // 스레드명을 변경할 대상 패턴
    private static final Pattern[] TARGET_PATTERNS = {
        Pattern.compile(".*")  // 🔴 TEMP: 모든 스레드에 적용 (테스트용)
    };

    // 이니셜 (A-Z)
    private static final String[] INITIALS = {
        "SRV", "API", "WEB", "DBG", "TST", "DEV", "PRD", "STG"
    };

    /**
     * 현재 스레드의 이름이 변경 대상인지 확인
     */
    public static boolean shouldRenameThread() {
        String currentName = Thread.currentThread().getName();
        for (Pattern pattern : TARGET_PATTERNS) {
            if (pattern.matcher(currentName).matches()) {
                return true;
            }
        }
        return false;
    }

    /**
     * 현재 스레드의 이름을 변경 (실제 추적 컨텍스트에서 TraceId/SpanId 읽기)
     *
     * 형식: {TraceId 7자}{SpanId 6자}{이니셜}{숫자}
     * 예: 26cd7480d92c2DBG6176
     */
    public static void renameThread() {
        renameThread(Span.current());
    }

    /**
     * 현재 스레드의 이름을 변경 (Span에서 SpanContext 직접 읽기)
     *
     * @param span Span 객체 (null 가능)
     */
    public static void renameThread(Span span) {
        String currentName = Thread.currentThread().getName();

        // Span에서 SpanContext 가져오기
        if (span == null || !span.getSpanContext().isValid()) {
            return;
        }

        SpanContext context = span.getSpanContext();
        String traceId = context.getTraceId();
        String spanId = context.getSpanId();

        if (traceId == null || spanId == null) {
            return;
        }

        // 포맷: {TraceId 7자}{SpanId 6자}{이니셜}{숫자}
        String newThreadName = formatThreadName(traceId, spanId);

        // 스레드명 변경
        try {
            Thread.currentThread().setName(newThreadName);
            // 로그에 TraceId와 SpanId 출력
            System.out.println("===============================================");
            System.out.println("[ThreadName] RENAMED THREAD");
            System.out.println("[ThreadName] Original Name: " + currentName);
            System.out.println("[ThreadName] New Name:      " + newThreadName);
            System.out.println("[ThreadName] TraceId:       " + traceId);
            System.out.println("[ThreadName] SpanId:        " + spanId);
            System.out.println("===============================================");
        } catch (Exception e) {
            System.err.println("[ThreadName] Failed to rename thread: " + e.getMessage());
        }
    }

    /**
     * TraceId와 SpanId로 형식화된 스레드명 생성
     * 형식: {TraceId 7자}{SpanId 6자}{이니셜}{숫자}
     * 예: 26cd7480d92c2DBG6176
     */
    private static String formatThreadName(String traceId, String spanId) {
        // TraceId 7자 (앞 7자)
        String tracePrefix = traceId.substring(0, Math.min(7, traceId.length()));

        // SpanId 6자 (앞 6자)
        String spanPrefix = spanId.substring(0, Math.min(6, spanId.length()));

        // 이니셜 (무작위 선택)
        String initial = INITIALS[Math.abs(traceId.hashCode()) % INITIALS.length];

        // 숫자 (0-9999)
        int number = ThreadLocalRandom.current().nextInt(10000);

        return String.format("%s%s%s%04d", tracePrefix, spanPrefix, initial, number);
    }

    /**
     * 현재 스레드명이 변경 대상 패턴인지 확인
     */
    public static boolean isTargetThread() {
        return shouldRenameThread();
    }
}
```

**주요 변경사항:**
- ✅ SpanContext에서 TraceId/SpanId 추출
- ✅ 형식화된 스레드명 생성 (7자 + 6자 + 3자 + 4자)
- ✅ Thread.currentThread().setName() 호출
- ✅ 대상 스레드 패턴 매칭 (현재는 모든 스레드 대상)

```java
public static void onMethodExit(@Advice.Enter SpanHelper spanHelper) {
    System.err.println("<<< ADVICE EXIT");

    if (spanHelper != null && spanHelper.isValid()) {
        spanHelper.complete();
        System.err.println("<<< SPAN COMPLETED");

        // ===== 🆕 스레드명 복원 =====
        String originalThreadName = spanHelper.getOriginalThreadName();
        if (originalThreadName != null) {
            try {
                String currentName = Thread.currentThread().getName();
                Thread.currentThread().setName(originalThreadName);
                System.out.println("===============================================");
                System.out.println("[ThreadName] RESTORED THREAD");
                System.out.println("[ThreadName] From: " + currentName);
                System.out.println("[ThreadName] To:   " + originalThreadName);
                System.out.println("===============================================");
            } catch (Exception e) {
                System.err.println("<<< Failed to restore thread name: " + e.getMessage());
            }
        }
        // =============================
    }
}
```

### 5. ByteBuddyAgent 수정 (이미 로드된 클래스 체크)

**파일**: `bytebuddy-agent/src/main/java/com/javaagent/bytebuddy/ByteBuddyAgent.java`

**변경사항:**
```java
private static boolean injectHelper(ClassLoader targetLoader) {
    if (targetLoader == null) {
        System.err.println("[ByteBuddy] Target ClassLoader is null, cannot inject helper");
        return false;
    }

    try {
        for (String helperClassName : AgentConstants.HELPER_CLASSES) {
            // 🆕 이미 로드된 클래스인지 체크
            try {
                targetLoader.loadClass(helperClassName);
                System.out.println("[ByteBuddy] Helper already loaded: " + helperClassName + ", skipping injection");
                continue;
            } catch (ClassNotFoundException e) {
                // Class not loaded, proceed with injection
            }

            // Load helper class bytecode from System ClassLoader
            byte[] classBytes = loadHelperClassBytes(helperClassName);
            if (classBytes == null) {
                System.err.println("[ByteBuddy] Failed to load bytecode for " + helperClassName);
                continue;
            }

            // Inject into target ClassLoader using reflection
            injectClassIntoLoader(targetLoader, helperClassName, classBytes);
            System.out.println("[ByteBuddy] Injected helper: " + helperClassName + " into ClassLoader: " + targetLoader);
        }

        return true;

    } catch (Exception e) {
        System.err.println("[ByteBuddy] Failed to inject helper classes: " + e.getMessage());
        e.printStackTrace();
        return false;
    }
}
```

## 🔥 발생한 문제점

### 문제 1: ThreadName 로그가 전혀 출력되지 않음

**증상:**
```
>>> ADVICE ENTER: public java.lang.String com.test.service.test1.controller.Test1Controller.test1(java.lang.String)
[SpanHelper] Span created and made current: ...
>>> SPAN CREATED: true
>>> Set attribute: userId = THREAD_NAME_FORMAT_TEST_1774616915
<<< ADVICE EXIT

# 예상되었던 로그:
>>> [ThreadName] Check: spanHelper=true, isTarget=true
===============================================
[ThreadName] RENAMED THREAD
...
```

**원인 분석:**
1. Advice 자체는 실행됨 (`>>> ADVICE ENTER`, `>>> SPAN CREATED` 확인)
2. 하지만 ThreadName 관련 로그가 전혀 없음
3. `Transformed bytecode created, size: 6458` - 항상 같은 크기

**결론:** 소스 코드 변경사항이 런타임에 반영되지 않음

### 문제 2: Helper injection 실패 (중복 클래스 정의)

**증상:**
```
[ByteBuddy] Failed to inject helper classes: null
java.lang.reflect.InvocationTargetException
Caused by: java.lang.LinkageError: loader 'app' attempted duplicate class definition for com.javaagent.bytebuddy.advices.SpanAdvice
```

**원인:**
1. SpanAdvice가 이미 agent JAR에서 로드되어 있음
2. HELPER_CLASSES에 SpanAdvice가 포함되어 있어서 다시 injection 시도
3. 중복 정의 에러 발생

### 문제 3: JAR 파일 손상

**증상:**
```
[ByteBuddy] Error loading helper bytecode for com.javaagent.bytebuddy.advices.SpanAdvice: ZipFile invalid LOC header (bad signature)
```

**원인:**
1. `cp` 명령어로 JAR 파일 복사 시 파일이 손상됨
2. Windows 환경에서 파일 사용 중인 경우 복사 실패

### 문제 4: Maven 로컬 리포지토리 미설치

**증상:**
```
# 파일 시간 확인
agent-commons의 AgentConstants.class: 22:40 (최신!)
bytebuddy-agent JAR의 AgentConstants.class: 20:01 (옛날 버전!)
```

**로그 확인:**
```
[ByteBuddy] Helper already loaded: com.javaagent.bytebuddy.advices.SpanAdvice
[ByteBuddy] Helper already loaded: com.javaagent.bytebuddy.helper.SpanAttributeHelper
[ByteBuddy] Helper already loaded: com.javaagent.bytebuddy.helper.SpanHelper
[ByteBuddy] Helper already loaded: com.javaagent.bytebuddy.helper.JaegerLinkLookupService
# ❌ ThreadNameHelper가 목록에 없음!
```

**원인:**
1. `mvn package`만 실행하여 로컬 리포지토리에 설치 안 됨
2. bytebuddy-agent가 Maven 로컬 리포지토리(`~/.m2/repository`)에서 agent-commons를 가져옴
3. 그곳에 있는 게 예전 버전임

### 문제 5: inline=true로 bytecode hard-coded

**증상:**
```
[ByteBuddy] Transformed bytecode created, size: 6458
```
- 항상 같은 크기의 bytecode가 생성됨
- inline=false로 변경했는데도 변화 없음

**원인:**
1. bytebuddy-advice도 `mvn package`만 해서 로컬 리포지토리에 설치 안 됨
2. bytebuddy-agent가 예전 버전의 SpanAdvice를 가져옴
3. inline=true로 설정된 예전 bytecode가 그대로 사용됨

```bash
cd agent-commons && mvn clean install
cd bytebuddy-advice && mvn clean install
cd bytebuddy-agent && mvn clean package
# → 로컬 리포지토리에 최신 버전 설치
```

**이유:**
- `mvn package`는 프로젝트를 빌드하지만 로컬 리포지토리(`~/.m2/repository`)에 설치하지 않음
- 다중 모듈 프로젝트에서는 의존성 모듈을 먼저 `install` 해야 함
- 그렇지 않으면 다른 모듈이 로컬 리포지토리에 있는 예전 버전을 가져옴

### 해결책 2: inline=false로 변경 (개발용)

**문제:**
- `inline=true`로 설정되어 있어서 Advice 코드가 bytecode에 직접 복사됨
- 소스 수정이 반영되지 않음

**해결:**
```java
// 변경 전
@Advice.OnMethodEnter(inline = true)
@Advice.OnMethodExit(inline = true)

// 변경 후 (개발용)
@Advice.OnMethodEnter(inline = false)
@Advice.OnMethodExit(inline = false)
```

**이유:**
- `inline=true`: Advice 코드를 target 메서드의 bytecode에 직접 복사
  - 장점: 메서드 호출 오버헤드 없음
  - 단점: 소스 수정 반영 안 됨, 디버깅 어려움

- `inline=false`: Advice 코드를 메서드 호출 방식으로 실행
  - 장점: 소스 수정 즉시 반영, 디버깅 쉬움
  - 단점: 메서드 호출 1회 추가 (성능 차이 미미함)

### 해결책 3: HELPER_CLASSES에서 SpanAdvice 제거

**문제:**
- SpanAdvice가 이미 agent JAR에 포함되어 있는데 다시 injection 시도
- 중복 클래스 정의 에러 발생

**해결:**
```java
// AgentConstants.java
public static final String[] HELPER_CLASSES = {
    // SpanAdvice 제거 (이미 agent JAR에 있음)
    "com.javaagent.bytebuddy.helper.SpanAttributeHelper",
    "com.javaagent.bytebuddy.helper.SpanHelper",
    "com.javaagent.bytebuddy.helper.JaegerLinkLookupService",
    "com.javaagent.bytebuddy.helper.ThreadNameHelper",  // 추가
    "com.javaagent.bytebuddy.advices.KernelAdvice",
    "com.javaagent.bytebuddy.helper.KernelHelper"
};
```

### 해결책 4: 이미 로드된 클래스 체크 로직 추가

**문제:**
- 이미 로드된 클래스를 다시 injection 시도
- 불필요한 작업과 잠재적 오류

**해결:**
```java
// ByteBuddyAgent.java
for (String helperClassName : AgentConstants.HELPER_CLASSES) {

        continue;
    } catch (ClassNotFoundException e) {
        // Class not loaded, proceed with injection
    }

    // ... injection 로직 ...
}
```

## 💡 배운 점

### 1. Maven 다중 모듈 프로젝트에서는 mvn install 필수

**잘못된 방법:**
```bash
cd module-a && mvn clean package
cd module-b && mvn clean package  # module-a를 의존하는 모듈
# → module-b가 로컬 리포지토리의 예전 버전을 가져옴
```

**올바른 방법:**
```bash
cd module-a && mvn clean install
cd module-b && mvn clean package
# → module-b가 최신 버전의 module-a를 가져옴
```

### 2. inline=true vs inline=false

| 측면 | inline=true | inline=false |
|------|-------------|--------------|
| **실행 방식** | Advice 코드를 bytecode에 직접 복사 | 메서드 호출 방식 |
| **성능** | 메서드 호출 오버헤드 없음 | 메서드 호출 1회 추가 |
| **소스 수정 반영** | ❌ 안 됨 (hard-coded) | ✅ 즉시 반영 |
| **디버깅** | ❌ 어려움 | ✅ 쉬움 |
| **추천 용도** | 프로덕션 (성능) | 개발 (편의성) |

### 3. 빌드 순서와 타임스탬프 확인

**빌드 순서:**
```
1. agent-commons (의존성 없음)
   → mvn clean install

2. bytebuddy-advice (agent-commons 의존)
   → mvn clean install

3. bytebuddy-agent (agent-commons, bytebuddy-advice 의존)
   → mvn clean package
```

**타임스탬프 확인:**
```bash
# JAR 파일 내의 클래스 타임스탬프 확인
unzip -l bytebuddy-agent-1.0.0.jar | grep AgentConstants

# 결과 예시
1935  2026-03-27 22:55   com/javaagent/commons/AgentConstants.class
```

### 4. ByteBuddy Helper Injection 주의사항

- 이미 agent JAR에 포함된 클래스는 다시 injection하지 않아야 함
- HELPER_CLASSES 목록에서 제외하거나 이미 로드된 클래스인지 체크해야 함
- 중복 injection 시 `LinkageError: attempted duplicate class definition` 에러 발생

### 5. Java 모듈 시스템 (--add-opens)

Java 9+에서는 리플렉션으로 내부 API 접근 시 제한이 있음:

```xml
<jvmArguments>
    --add-opens java.base/java.lang=ALL-UNNAMED
    --add-opens java.base/java.lang.reflect=ALL-UNNAMED
    --add-opens java.base/java.io=ALL-UNNAMED
</jvmArguments>
```

하지만 이번 문제의 근본 원인은 모듈 시스템 제약이 아니었습니다.
