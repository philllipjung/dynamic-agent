## 시스템 개요

### 아키텍처

```
┌─────────────────┐      ┌──────────────────┐
│  Target JVM     │◄─────┤ ByteBuddy Agent  │
│  (test1-service)│      │  (bytebuddy-agent)│
└────────┬────────┘      └─────────┬────────┘
         │                         │
         │                         │
         ▼                         ▼
┌─────────────────┐      ┌──────────────────┐
│   Arthas Agent  │      │ Agent Management │
│ (arthas-manager)│      │   (agent-server) │
└────────┬────────┘      └─────────┬────────┘
         │                         │
         │                         │
         ▼                         ▼
    ┌──────────────────────────────────┐
    │            Redis                 │
    │  (상태 저장, 이벤트 로그)         │
    └──────────────────────────────────┘
```

### 주요 기능

- **런타임 계측 (Runtime Instrumentation)**: 실행 중인 JVM에 코드 변경 없이 기능 추가
- **OpenTelemetry 통합**: Span 생성, 속성 추가
- **상태 관리**: Redis를 통한 영구 저장
- **이벤트 로그**: 모든 작업의 히스토리 추적

## 모듈 구조

### 1. bytebuddy-agent

**역할:** JVM 부착 및 바이트코드 조작

**주요 클래스:**
- `ByteBuddyAgent` - 메인 에이전트 클래스
- `AttachManager` - JVM 부착 관리

**기능:**
```java
// Span 생성
ByteBuddyAgent.createSpan(pid, className, methodName)

// Span Attribute 생성
ByteBuddyAgent.createSpanAttributeAdvice(pid, className, methodName, paramMapping)

// Event 생성
ByteBuddyAgent.createEventAdvice(className, methodName)
```

**주요 포인트:**
- `premain()` - JVM 시작 시 로드 (`-javaagent`)
- `agentmain()` - 실행 중인 JVM에 부착 (`Attach API`)
- HTTP 서버 (포트 9999-10010) - 원격 제어

---

### 2. bytebuddy-advice

**역할:** 계측 코드 (Advice 클래스들)

**주요 클래스:**
- `SpanAdvice` - OpenTelemetry Span 생성
- `EventAdvice` - HTTP 요청/응답 캡처
- `SpanAttributeHelper` - Span 속성 설정 헬퍼

**Advice 적용 방식:**
```java
@Advice.OnMethodEnter
public static void onMethodEnter(
    @Advice.Argument(0) String arg1,
    @Advice.Local("span") Span span
) {
    // 메서드 진입 시 실행될 코드
}

@Advice.OnMethodExit
public static void onMethodExit(@Advice.Local("span") Span span) {
    // 메서드 종료 시 실행될 코드
}
```

---

### 3. arthas-manager

**역할:** Alibaba Arthas 통합 관리

**주요 클래스:**
- `ArthasManager` - Arthas attach/detach, 커맨드 실행

**기능:**
```java
// Arthas attach
ArthasManager.attachByPid(pid)
ArthasManager.attachByClassName(className)

// 명령 실행
ArthasManager.trace(className, methodName)
ArthasManager.watch(className, methodName, expression, limit)
ArthasManager.stack(className, methodName)

// HTTP API 통신
ArthasManager.executeTunnelApiCommand(command)
```

**Arthas HTTP API:**
```
POST http://127.0.0.1:8563/api
{
  "action": "exec",
  "command": "watch com.example.MyClass myMethod params -n 1 -x 1",
  "execTimeout": "5000"
}
```

### 4. agent-server

**역할:** REST API 서버, 상태 관리

**주요 컨트롤러:**
- `ByteBuddyController` - ByteBuddy 작업 API
- `ArthasController` - Arthas 작업 API
- `AgentManagementController` - 상태 조회/관리 API

**주요 서비스:**
- `AgentStateService` - Redis 상태 관리
- `ParameterMappingService` - 파라미터 매핑 관리

**Redis 키 구조:**
```
jvm:{pid}                              → JVM 상태
instrumentation:{pid}:{class}:{method} → Instrumentation 상태
jvms:all                               → 모든 PID SET
events:all                             → 전체 이벤트 LIST
events:{pid}                           → PID별 이벤트 LIST
```

## 예상되는 문제점

### 1. 클래스로더 문제

**현상:**
```
ClassNotFoundException: com.javaagent.bytebuddy.advices.SpanAdvice
```

**원인:**
- Advice 클래스가 타겟 JVM의 클래스로더에 로드되지 않음
- System ClassLoader와 타겟 ClassLoader 사이의 격리

**해결:**
- `injectHelper()` 메서드로 리플렉션 통해 클래스 주입
- 타겟 ClassLoader에서 Advice 클래스 로드

### 2. 파라미터 이름 유지 문제

**현상:**
```
IllegalArgumentException: Name for argument of type [java.lang.String] not specified
```

**원인:**
- Java 컴파일러가 기본적으로 파라미터 이름을 바이트코드에 저장하지 않음
- Spring의 `@RequestParam`이 이름을 찾지 못함

**해결:**
- `@RequestParam("pid")` 처럼 명시적으로 이름 지정
- 또는 Maven 설정에 `<parameters>true</parameters>` 추가

### 3. Redis 연결 문제

**현상:**
```
Unable to connect to Redis: localhost:6379
```

**원인:**
- Redis가 실행 중이 아님
- 잘못된 호스트/포트 설정

**해결:**
- Redis 실행 확인: `docker ps | grep redis`
- 설정 확인: `application.properties`

### 4. Attach 실패

**현상:**
```
com.sun.tools.attach.AttachNotSupportedException: Unable to attach
```

**원인:**
- 권한 부족 (Linux)
- PID가 잘못됨
- 타겟 JVM이 tools.jar를 포함하지 않음

**해결:**
- 같은 사용자로 실행
- `jps -l`로 PID 확인
- JDK 사용 (JDK가 아닌 JRE는 attach 불가)

### 5. Port 충돌

**현상:**
```
Port 8080 was already in use
```

**원인:**
- 이미 실행 중인 agent-server
- 다른 프로세스가 포트 점유

**해결:**
```bash
# 포트 점유 프로세스 확인
netstat -ano | findstr :8080

# 프로세스 종료
taskkill /F /PID <PID>
```

### 6. 재변환 (Retransformation) 실패

**현상:**
```
class retransformation failed: unsupported class version
```

**원인:**
- 이미 변환된 클래스를 다시 변환 시도
- 클래스 버전 불일치

**해결:**
- 재변환 전에 상태 확인
- 변환 이력 추적

### 7. 메모리 누수

**현상:**
- JVM 메모리 지속 증가
- Redis 메모리 과다 사용

**원인:**
- Instrumentation 계속 추가
- 이벤트 로그 무한 증가
- ClassLoader 누수

**해결:**
- 이벤트 로그 자동修剪 (최대 1000개)
- TTL 설정
- 주기적으로 detach

## 해결방안

### 1. 클래스로더 문제 해결

**코드:**
```java
private static boolean injectHelper(ClassLoader targetLoader) {
    for (String helperClassName : AgentConstants.HELPER_CLASSES) {
        byte[] classBytes = loadHelperClassBytes(helperClassName);
        injectClassIntoLoader(targetLoader, helperClassName, classBytes);
    }
    return true;
}

private static void injectClassIntoLoader(ClassLoader targetLoader,
                                          String className,
                                          byte[] classBytes) throws Exception {
    Method defineClass = ClassLoader.class.getDeclaredMethod(
        "defineClass", String.class, byte[].class, int.class, int.class
    );
    defineClass.setAccessible(true);
    defineClass.invoke(targetLoader, className, classBytes, 0, classBytes.length);
}
```

### 2. 상태 동기화

**문제:** 메모리 상태와 Redis 불일치

**해결:**
- `@PostConstruct`에서 Redis 상태 복구
- 주기적으로 상태 검증
```java
@Scheduled(fixedRate = 60000)
public void validateState() {
    // Redis와 실제 attach 상태 비교
}
```

### 3. 에러 처리 개선

```java
@PostMapping("/attach")
public Map<String, Object> attach(@RequestBody AttachRequest request) {
    try {
        // ... attach 로직
        return Map.of("success", true, "message", result);
    } catch (AttachNotSupportedException e) {
        return Map.of(
            "success", false,
            "error", "ATTACH_NOT_SUPPORTED",
            "message", "JDK가 필요합니다. JRE는 attach를 지원하지 않습니다."
        );
    } catch (IOException e) {
        return Map.of(
            "success", false,
            "error", "PID_NOT_FOUND",
            "message", "PID " + pid + "를 찾을 수 없습니다."
        );
    }
}
```

### 4. 건강상태 확인 (Health Check)

```java

## 트러블슈팅 가이드

### AgentStateService 기반 진단 시스템

`AgentStateService`는 Redis에 저장된 상태와 이벤트 로그를 통해 문제를 진단하고 해결하는 핵심 컴포넌트입니다.

#### 진단 API 활용

**1. 상태 스냅샷 가져오기**

```bash
# 전체 JVM 상태 확인
curl http://localhost:8080/api/agent/jvms | jq '.'
```

응답 분석:
```json
{
  "success": true,
  "jvms": [
    {
      "pid": "11248",
      "className": "com.test.service.test1.Test1Application",
      "agentType": "BYTEBUDDY",  // 또는 "ARTHAS", "BOTH"
      "attachTime": "2026-03-29T17:05:54.3417089"
    }
  ],
  "count": 1
}
```

**진단 포인트:**
- `count`가 0이면: Redis에 저장된 상태 없음 → attach 필요
- `agentType`이 예상과 다르면: 일부만 attach됨
- `attachTime`이 너무 오래됐으면:僵尸 상태일 가능성

---

**2. 이벤트 로그 분석**

```bash
# 전체 이벤트 조회 (최신순)
curl "http://localhost:8080/api/agent/events?limit=50" | jq '.events[]'

# 특정 PID 이벤트 조회
curl "http://localhost:8080/api/agent/events?pid=11248&limit=20" | jq '.events[]'
```

이벤트 타입 분석:
```json
[
  {"type": "attach", "pid": "11248", "agentType": "BYTEBUDDY", "timestamp": "..."},
  {"type": "attach", "pid": "11248", "agentType": "ARTHAS", "timestamp": "..."},
  {"type": "instrumentation", "pid": "11248", "className": "...", "methodName": "...", "adviceType": "SPAN", "timestamp": "..."},
  {"type": "detach", "pid": "11248", "agentType": "BOTH", "timestamp": "..."}
]
```

**진단 포인트:**
- `attach` 없이 `instrumentation`만 있으면: 순서 오류
- `detach` 후에도 `instrumentation` 이벤트가 계속되면: 정리 안 됨
- 같은 `type`의 이벤트가 반복되면: 재시도 실패

---

**3. Instrumentation 상태 확인**

```bash
# 특정 JVM의 instrumentation 목록
curl http://localhost:8080/api/agent/instrumentations/11248 | jq '.'
```

응답 분석:
```json
{
  "success": true,
  "pid": "11248",
  "instrumentations": [
    {
      "className": "com.test.service.UserService",
      "methodName": "getUserId",
      "adviceType": "SPAN_ATTRIBUTE",
      "appliedAt": "2026-03-29T17:10:15.123456"
    }
  ],
  "count": 1
}
```

**진단 포인트:**
- `count`가 0이면: instrumentation 적용 안 됨
- `appliedAt`이 `attachTime`보다 이전이면: 데이터 불일치

### 시나리오별 진단 및 해결

#### 시나리오 1: "Already attached" but not working

**현상:**
```bash
curl -X POST http://localhost:8080/api/bytebuddy/attach -d '{"pid": "11248"}'
# 응답: {"success": true, "message": "Already attached to PID 11248"}

# 하지만 실제로는 동작 안 함
```

**진단:**

Step 1: Redis 상태 확인
```bash
curl http://localhost:8080/api/agent/jvms/11248 | jq '.jvm.agentType'
# 결과: "BYTEBUDDY"
```

Step 2: 실제 프로세스 확인
```bash
jps -l | grep 11248
# 결과: 프로세스 없음!
```

Step 3: 이벤트 로그 확인
```bash
curl "http://localhost:8080/api/agent/events?pid=11248&limit=5" | jq '.events[0]'
# 결과: {"type": "attach", ...} 이지만 detach 이벤트 없음
```

**원인:** Redis에는 상태가 있지만 실제 프로세스는 종료됨

**해결:**
```bash
# 1. Redis 상태 정리
curl -X DELETE "http://localhost:8080/api/bytebuddy/detach?pid=11248"

# 2. 실제 프로세스 확인 후 재시작
jps -l  # 새로운 PID 확인

# 3. 새로 attach
curl -X POST http://localhost:8080/api/bytebuddy/attach \
  -d '{"pid": "<NEW_PID>", "className": "..."}'
```

#### 시나리오 2: Instrumentation failed silently

**현상:**
```bash
curl -X POST http://localhost:8080/api/bytebuddy/createSpan \
  -d '{"pid": "11248", "className": "...", "methodName": "..."}'
# 응답: {"success": true, ...}

# 하지만 span이 생성되지 않음
```

**진단:**

Step 1: Instrumentation 상태 확인
```bash
curl http://localhost:8080/api/agent/instrumentations/11248 | jq '.instrumentations'
# 결과: [] (비어있음!)
```

Step 2: 이벤트 로그 확인
```bash
curl "http://localhost:8080/api/agent/events?pid=11248&limit=10" | jq '.events[] | select(.type == "instrumentation")'
# 결과: instrumentation 이벤트 없음
```

Step 3: JVM 상태 확인
```bash
curl http://localhost:8080/api/agent/jvms/11248 | jq '.jvm.agentType'
# 결과: "ARTHAS" (ByteBuddy가 attach 안 됨!)
```

**원인:** Arthas만 attach되어 있고 ByteBuddy가 attach 안 됨

**해결:**
```bash

curl -X POST http://localhost:8080/api/bytebuddy/attach \
  -d '{"pid": "11248", "className": "..."}'

# 결과: "BOTH" ✅

# 3. instrumentation 재시도
curl -X POST http://localhost:8080/api/bytebuddy/createSpan \
  -d '{"pid": "11248", "className": "...", "methodName": "..."}'

## 계측 충돌 감지 시스템

### 개요

OpenTelemetry Java Agent, ByteBuddy, Arthas 간의 계측 범위 겹침을 감지하고 충돌을 방지하는 시스템입니다.

### 충돌 시나리오

``┌─────────────────────────────────────────────────────────────┐
│  충돌 예시                                                   │
├─────────────────────────────────────────────────────────────┤
│  ❌ OpenTelemetry가 이미 계측:                             │
│     - @Controller의 createOrder()                           │
│     - @Repository의 findById()                              │
│                                                             │
│  ✅ ByteBuddy가 계측:                                       │
│     - @Service의 calculateDiscount()                        │
│     - @Service의 processPayment()                           │
└─────────────────────────────────────────────────────────────┘
```

### OpenTelemetry 자동 계측 범위

| 범위 | 예시 | ByteBuddy 계측 |
|------|------|----------------|
| **@Controller** | OrderController | ❌ 제외 |
| **@RestController** | ApiController | ❌ 제외 |
| **@Repository** | OrderRepository | ❌ 제외 |
| **JpaRepository** | UserRepository | ❌ 제외 |
| **RestTemplate** | HTTP Client | ❌ 제외 |
| **@Service** | OrderService | ✅ 안전 |
| **비즈니스 로직** | calculate* | ✅ 안전 |

### API 사용법

#### 1. 계측 전 충돌 확인 (자동)

`/api/bytebuddy/createSpan`과 `/api/bytebuddy/createSpanAttribute`는 **자동으로 충돌을 확인**합니다.

```bash
# 충돌이 있는 경우 계측 거부
curl -X POST http://localhost:8080/api/bytebuddy/createSpan \
  -H "Content-Type: application/json" \
  -d '{
    "pid": "11248",
    "className": "com.example.OrderController",
    "methodName": "createOrder"
  }'

# 응답:
{
  "success": false,
  "message": "CONFLICT: OpenTelemetry auto-instruments all @Controller methods",
  "conflictType": "OPENTELEMETRY",
  "recommendation": "Skip instrumentation, OpenTelemetry will cover this"
}
```

```bash
# 충돌이 없는 경우 계측 진행
curl -X POST http://localhost:8080/api/bytebuddy/createSpan \
  -H "Content-Type: application/json" \
  -d '{
    "pid": "11248",
    "className": "com.example.OrderService",
    "methodName": "calculateDiscount"
  }'

# 응답:
{
  "success": true,
  "message": "SUCCESS: Created span for com.example.OrderService.calculateDiscount"
}
```

### 2. ByteBuddy Detach - Instrumentation 유지 🟡

#### 현상
```java
// ByteBuddyAgent.java:790-798
public static String detach() {
    stopHttpServer();
    appliedMethods.clear();
    initialized.set(false);
    return "SUCCESS: Detached (instrumentation remains in target JVM)";
}
```

#### 실제 문제
- HTTP 서버만 중지
- 적용된 bytecode transformation은 **제거되지 않음**
- `ClassFileTransformer`가 제거되지 않음
- JVM이 재시작될 때까지 instrumentation이 유지됨

#### 해결 방안

**1. Transformer 제거 및 Retransformation**

```java
// ByteBuddyAgent.java 개선
private static final Map<String, ClassFileTransformer> activeTransformers = new ConcurrentHashMap<>();
private static final Map<String, Class<?>> transformedClasses = new ConcurrentHashMap<>();

public static String createSpan(String pid, String className, String methodName) {
    // ... 기존 코드 ...

    // Transformer 추적
    ClassFileTransformer transformer = new ClassFileTransformer() { ... };
    instrumentation.addTransformer(transformer, true);
    instrumentation.retransformClasses(targetClass);

    // 추적 맵에 저장
    String key = className + "." + methodName;
    activeTransformers.put(key, transformer);
    transformedClasses.put(key, targetClass);

    // ... 기존 코드 ...
}

public static String detach() {
    try {
        // 1. 모든 transformer 제거
        for (Map.Entry<String, ClassFileTransformer> entry : activeTransformers.entrySet()) {
            String key = entry.getKey();
            ClassFileTransformer transformer = entry.getValue();
            Class<?> targetClass = transformedClasses.get(key);

            if (transformer != null && instrumentation != null) {
                instrumentation.removeTransformer(transformer);

                // 원래 클래스로 복원 (retransform without transformer)
                try {
                    instrumentation.retransformClasses(targetClass);
                    System.out.println("[ByteBuddy] Restored original class: " + key);
                } catch (Exception e) {
                    System.err.println("[ByteBuddy] Failed to restore " + key + ": " + e.getMessage());
                }
            }
        }

        // 2. 정리
        activeTransformers.clear();
        transformedClasses.clear();
        stopHttpServer();
        appliedMethods.clear();
        initialized.set(false);

        return "SUCCESS: Detached and removed all transformations";
    } catch (Exception e) {
        return "ERROR: Failed to detach - " + e.getMessage();
    }
}
```

**2. 클래스 원본 백업 (선택 사항)**

```java
private static final Map<String, byte[]> originalClassBytes = new ConcurrentHashMap<>();

                }
                return null; // 변환하지 않음
            }
        };
        instrumentation.addTransformer(cachingTransformer, true);
        instrumentation.retransformClasses(targetClass);
        instrumentation.removeTransformer(cachingTransformer);
    }

    // ... 기존 instrumentation 코드 ...
}

**1. 주기적인 상태 검증**

// AgentStateService.java 개선
@Scheduled(fixedRate = 60000) // 1분마다
public void validateAndCleanupJvmStates() {
    try {
        List<AgentStateResponse.JvmState> jvms = getAllJvms();

        for (AgentStateResponse.JvmState jvm : jvms) {

            // 실제 프로세스 확인

                try {
                    ByteBuddyAgent.detach();
                } catch (Exception e) {
                    System.err.println("[AgentStateService] Failed to detach ByteBuddy: " + e.getMessage());
                }

                try {
                    ArthasManager.detach(pid);
                } catch (Exception e) {
                    System.err.println("[AgentStateService] Failed to detach Arthas: " + e.getMessage());
                }
            }
        }
    } catch (Exception e) {
        System.err.println("[AgentStateService] Validation failed: " + e.getMessage());

private boolean isProcessAlive(String pid) {
    try {
        // Unix/Linux/Mac
        String os = System.getProperty("os.name").toLowerCase();
        if (os.contains("win")) {
            // Windows
            ProcessBuilder pb = new ProcessBuilder("tasklist", "/FI", "\"PID eq " + pid + "\"");
            pb.redirectErrorStream(true);
            Process process = pb.start();

            BufferedReader reader = new BufferedReader(new InputStreamReader(process.getInputStream()));
            String line;
            while ((line = reader.readLine()) != null) {
                if (line.contains(pid)) {
                    return true;
                }
            }
            return false;
        } else {
            // Unix/Linux/Mac
            ProcessBuilder pb = new ProcessBuilder("ps", "-p", pid);
            pb.redirectErrorStream(true);
            Process process = pb.start();

            BufferedReader reader = new BufferedReader(new InputStreamReader(process.getInputStream()));
            String line;
            while ((line = reader.readLine()) != null) {
                if (line.contains(pid)) {
                    return true;
                }
            }
            return false;
        }
    } catch (Exception e) {
        System.err.println("[AgentStateService] Failed to check process " + pid + ": " + e.getMessage());
        return false;
    }
}

public void removeJvm(String pid, String agentType) {

**1. 주기적 정리 스케줄러**

```java
// ArthasManager.java 개선
static {
    // ... 기존 static 초기화 ...

    // 정리 스케줄러 시작
    ScheduledExecutorService cleanupScheduler = Executors.newSingleThreadScheduledExecutor();
    cleanupScheduler.scheduleAtFixedRate(() -> {
        cleanupOldWatchResults();
        cleanupOldStackResults();
        cleanupOldTraceResults();
    }, 5, 5, TimeUnit.MINUTES); // 5분마다 실행
}

private static void cleanupOldStackResults() {
    long fiveMinutesAgo = System.currentTimeMillis() - TimeUnit.MINUTES.toMillis(5);

    stackResults.entrySet().removeIf(entry -> {
        String jobId = entry.getKey();
        if (jobId.startsWith("stack-")) {
            long timestamp = Long.parseLong(jobId.substring(6));
            if (timestamp < fiveMinutesAgo) {
                stackStatus.remove(jobId);
                return true;
            }
        }
        return false;
    });
}

private static void cleanupOldTraceResults() {
    long fiveMinutesAgo = System.currentTimeMillis() - TimeUnit.MINUTES.toMillis(5);

    traceResults.entrySet().removeIf(entry -> {
        String jobId = entry.getKey();
        if (jobId.startsWith("trace-")) {
            long timestamp = Long.parseLong(jobId.substring(6));
            if (timestamp < fiveMinutesAgo) {
                traceStatus.remove(jobId);
                return true;
            }
        }
        return false;
    });
}
```

**원인 2: Scope 중첩 문제**

```java
// OpenTelemetry Java Agent가 이미 span 생성
@Controller
public class OrderController {
    @GetMapping("/orders")
    public String getOrders() {
        // OpenTelemetry가 자동으로 HTTP span 생성

        // ByteBuddy Advice가 또 span 생성
        @Advice.OnMethodEnter
        public static SpanHelper onMethodEnter() {
            SpanHelper spanHelper = SpanHelper.createSpanWithLinks(method, linkedContexts);

            // 문제: 두 개의 span이 서로 다른 context에 생성됨
            // 자식 span이 제대로 연결되지 않을 수 있음
            return spanHelper;
        }
    }
}
```

**영향:**
- Trace가 끊어짐 (fragmented trace)
- Parent-child 관계가 깨짐
- Jaeger/UI에서 연결되지 않은 독립된 span으로 표시됨

#### 해결 방안

**1. Context 수동 전파**

```java
// SpanAdvice.java 개선
@Advice.OnMethodEnter(inline = false)
public static SpanHelper onMethodEnter(
        @Advice.Origin String method,
        @Advice.This Object target,
        @Advice.AllArguments Object[] allArguments
) {
    // 1. 현재 OpenTelemetry context 캡처
    io.opentelemetry.context.Context currentContext = io.opentelemetry.context.Context.current();

    // 2. Span 생성 시 context 전달
    SpanHelper spanHelper = SpanHelper.createSpanWithLinks(method, currentContext);

    // 3. 새로운 span을 current로 만들기
    // SpanHelper.createSpanWithLinks 내부에서 이미 makeCurrent() 호출됨

    return spanHelper;
}

@Advice.OnMethodExit(inline = false)
public static void onMethodExit(@Advice.Enter SpanHelper spanHelper) {
    if (spanHelper != null && spanHelper.isValid()) {
        spanHelper.complete();  // 내부에서 scope.close() 호출됨
    }
}
```

**2. OpenTelemetry Context 직접 사용**

```java
// SpanHelper.java 개선
public static SpanHelper createSpanWithLinks(String methodName, io.opentelemetry.context.Context parentContext) {
    if (tracer == null) {
        System.err.println("[SpanHelper] Tracer not initialized!");
        return null;
    }

    // Parent context로 span builder 생성
    io.opentelemetry.api.trace.SpanBuilder builder = tracer.spanBuilder(methodName)
        .setParent(parentContext);  // ← 중요: parent context 설정

    // Links 추가
    if (linkedContexts != null && !linkedContexts.isEmpty()) {
        for (io.opentelemetry.api.trace.SpanContext context : linkedContexts) {
            builder.addLink(context);
        }
    }

    Span span = builder.startSpan();
    Scope scope = span.makeCurrent();

    return new SpanHelper(span, scope);
}
```

### 3. 부작용 (Side Effects)

#### 3.1 성능 저하

**문제:**
- 동일한 메서드에 3개의 도구가 동시에 적용
- 메서드 호출마다 3번의 overhead

```java
// 단일 메서드 호출 시

    // 2. ByteBuddy Advice: 진입 로직 (~1μs)
    // 3. Arthas watch: 파라미터 캡처 (~5μs)

    // === 실제 비즈니스 로직 ===

    // 4. Arthas watch: 결과 출력 (~5μs)
    // 5. ByteBuddy Advice: 종료 로직 (~1μs)

```

**2. 샘플링**

#### 3.2 ClassLoader Memory Leak

**문제:**
- 각 도구가 별도의 ClassLoader 사용
- 클래스가 unload되지 않음

```java
// ByteBuddy가 클래스를 주입할 때마다
ClassLoader targetLoader = targetClass.getClassLoader();
injectClassIntoLoader(targetLoader, "com.javaagent.bytebuddy.advices.SpanAdvice", classBytes);
// → targetLoader에 클래스가 영구적으로 로드됨
// → JVM이 종료될 때까지 메모리 유지
```

**영향:**
- PermGen/Metaspace 공간 부족
- OutOfMemoryError: Metaspace

#### 해결 방안

**1. 클래스 주입 전 중복 확인**

```java
// ByteBuddyAgent.java 개선
private static boolean injectHelper(ClassLoader targetLoader) {
    if (targetLoader == null) {
        return false;
    }

    try {
        for (String helperClassName : AgentConstants.HELPER_CLASSES) {
            // 이미 로드되어 있으면 skip
            try {
                targetLoader.loadClass(helperClassName);
                System.out.println("[ByteBuddy] Helper already loaded: " + helperClassName);
                continue;  // 중복 주입 방지
            } catch (ClassNotFoundException e) {
                // 로드되지 않았으므로 주입 진행
            }

            // ... 주입 코드 ...
        }
        return true;
    } catch (Exception e) {
        System.err.println("[ByteBuddy] Failed to inject helper: " + e.getMessage());
        return false;
    }
}
```

**2. WeakReference 사용**

```java
// ByteBuddyAgent.java 개선
private static final Map<ClassLoader, Set<String>> injectedLoaders = new WeakHashMap<>();

private static boolean injectHelper(ClassLoader targetLoader) {
    // WeakHashMap이므로 ClassLoader가 GC되면 자동 제거됨
    Set<String> injectedClasses = injectedLoaders.computeIfAbsent(
        targetLoader, k -> ConcurrentHashMap.newKeySet()
    );

    for (String helperClassName : AgentConstants.HELPER_CLASSES) {
        if (injectedClasses.contains(helperClassName)) {
            continue;  // 이미 주입됨
        }

        // 주입 후 기록
        injectClassIntoLoader(targetLoader, helperClassName, classBytes);
        injectedClasses.add(helperClassName);
    }

    return true;
}
```

#### 3.3 Thread Name 변경으로 인한 모니터링 실패

**문제:**
- ThreadNameHelper가 스레드명을 변경
- 로그 분석, 모니터링 시스템에서 혼란

```
# 원본 스레드명
http-nio-8081-exec-1

# ThreadNameHelper 변경 후
http-nio-8081-exec-1-traceid=abc123-spanid=def456

# 문제:
# - APM 도구가 스레드를 인식하지 못함
# - 로그 집계 시 스레드명이 달라져서 grouping 실패
```

#### 해결 방안

**1. MDC (Mapped Diagnostic Context) 사용**

```java
// ThreadNameHelper.java 개선
public static void renameThread(Span span) {
    if (!isTargetThread()) {
        return;
    }

    // 스레드명 대신 MDC 사용
    String originalThreadName = Thread.currentThread().getName();
    String traceId = span.getSpanContext().getTraceId();
    String spanId = span.getSpanContext().getSpanId();

    // MDC에 저장 (SLF4J/Logback이 자동으로 로그에 포함)
    org.slf4j.MDC.put("traceId", traceId);
    org.slf4j.MDC.put("spanId", spanId);

    // 스레드명은 변경하지 않음
    // Thread.currentThread().setName(...);  // ← 주석 처리

    System.out.println("[ThreadName] MDC updated: traceId=" + traceId + ", spanId=" + spanId);
}
```

**2. logback.xml 설정**

```xml
<configuration>
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - traceId=%X{traceId:-N/A} spanId=%X{spanId:-N/A} %msg%n</pattern>
        </encoder>
    </appender>
</configuration>
```
