### 2.2 워크플로우

⚠️ **중요**: Span Link를 생성하려면 **소스와 타겟 양쪽 모두**에 instrumentation을 적용해야 합니다!

```
===== Phase 1: 타겟 서비스 설정 =====
1. Service 선택 (test2-service) → PID 자동 감지
2. Target Method 설정
   ↓
3. [Watch로 파라미터 확인] 클릭
   → Arthas Watch 실행
   → 파라미터 리스트 표시
   → 체크박스로 선택 (userId)
   ↓
4. [Apply Instrumentation] 클릭
   → /api/bytebuddy/attach 호출 (test2 서비스)

===== Phase 2: 소스 서비스 설정 =====
5. Service 선택 (test1-service) → PID 자동 감지
6. Source Method 설정
   ↓
7. [Watch로 파라미터 확인] 클릭
   → 파라미터 리스트 표시
   → 체크박스로 선택 (userId - 동일한 속성명)
   ↓
8. [Apply Instrumentation] 클릭
   → /api/bytebuddy/attach 호출 (test1 서비스)

===== Phase 3: 링크 생성 테스트 =====
9. test2 요청 (먼저) → Span 생성
10. 10초 대기 (Jaeger 인덱싱)
11. test1 요청 (나중에) → 링크 생성!
```

**핵심 포인트:**
| 항목 | 설명 |
|------|------|
| **양쪽 적용** | test1, test2 **모두**에 instrumentation 적용 필요 |
| **동일 속성** | 양쪽 모두 **같은 속성 이름(userId)** 사용 |
| **실행 순서** | test2 → (10초 대기) → test1 |
| **링크 방향** | test1 → test2 (FOLLOWS_FROM) |

### 2.4 API 호출

⚠️ **Span Link 생성 시 양쪽 서비스 모두 호출 필요**

```javascript
// ===== 타겟 서비스 (test2) - 먼저 실행될 쪽 =====
POST /api/bytebuddy/attach
{
  "pid": "67890",  // test2 PID
  "instrumentation": "com.test.service.test2.controller.Test2Controller:test2:spanAttribute:userId"
}

// ===== 소스 서비스 (test1) - 나중에 실행될 쪽 =====
POST /api/bytebuddy/attach
{
  "pid": "12345",  // test1 PID
  "instrumentation": "com.test.service.test1.controller.Test1Controller:test1:spanAttribute:userId"
}

// Watch 파라미터 확인
POST /api/arthas/startWatch
{
  "className": "com.test.service.test1.controller.Test1Controller",
  "methodName": "test1",
  "limit": 1
}

// Watch 결과 조회
GET /api/arthas/getWatchResult/watch-{jobId}
```

**전체 시퀀스 예시:**
```bash
# 1. test2 적용
curl -X POST "http://localhost:8080/api/bytebuddy/attach" \
  -H "Content-Type: application/json" \
  -d '{"pid":"67890","instrumentation":"Test2Controller:test2:spanAttribute:userId"}'

# 2. test1 적용
curl -X POST "http://localhost:8080/api/bytebuddy/attach" \
  -H "Content-Type: application/json" \
  -d '{"pid":"12345","instrumentation":"Test1Controller:test1:spanAttribute:userId"}'

# 3. test2 요청 (먼저)
TEST_ID="LINK_$(date +%s)"
curl -H "userId: $TEST_ID" http://localhost:8082/test2

# 4. 10초 대기
sleep 10

# 5. test1 요청 (나중에, 링크 생성!)
curl -H "userId: $TEST_ID" http://localhost:8081/test1
```

## 6. 데이터 흐름

### 6.1 Span & Link 생성 흐름 (양쪽 서비스 적용)

```
===== 타겟 서비스 (test2) 설정 =====
사용자 입력 (test2-service 선택)
    ↓
[Watch로 파라미터 확인] → userId 감지
    ↓
[Apply Instrumentation] → test2에 적용
    ↓
ByteBuddy Agent Attach (PID: 67890)
    ↓
test2 instrumentation 완료

===== 소스 서비스 (test1) 설정 =====
사용자 입력 (test1-service 선택)
    ↓
[Watch로 파라미터 확인] → userId 감지 (동일한 속성명)
    ↓
[Apply Instrumentation] → test1에 적용
    ↓
ByteBuddy Agent Attach (PID: 12345)
    ↓
test1 instrumentation 완료

===== 링크 생성 테스트 =====
test2 요청 (userId=LINK_TEST_123)
    ↓
test2 Span 생성 → Jaeger에 저장
    ↓
10초 대기 (Jaeger 인덱싱)
    ↓
test1 요청 (userId=LINK_TEST_123)
    ↓
test1 Span 생성
    ↓
JaegerLinkLookupService 검색
    → 같은 userId를 가진 test2 span 찾음
    ↓
FOLLOWS_FROM 링크 생성! ✅
```

**중요한 점:**
| 단계 | 서비스 | 목적 |
|------|--------|------|
| 1 | test2 | 먼저 Span을 생성하여 링크의 대상이 됨 |
| 2 | (대기) | Jaeger 인덱싱 시간 (10초) |
| 3 | test1 | 나중에 Span을 생성하며 test2를 링크로 연결 |

### 6.2 Event Capturing 흐름

```
[Start Event Capturing]
    ↓
Event Advice 적용
    ↓
HTTP 요청 캡처 시작
    ↓
콘솔 출력 → WebSocket/SSE
    ↓
실시간 UI 표시
```

### 6.3 Arthas 분석 흐름

```
[Watch/Stack/Trace 시작]
    ↓
비동기 작업 시작 (jobId 반환)
    ↓
메서드 호출 대기
    ↓
주기적 상태 확인 (폴링)
    ↓
결과 완료 시 UI 표시
```
