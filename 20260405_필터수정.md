# 20260405 필터 계측 수정 - EventAdvice 네이밍 변경 및 아키텍처 이해

## 개요

`KernelAdvice`를 `EventAdvice`로 네이밍 변경하고, 필터 계측 기능을 테스트하면서 발생한 문제들을 해결한 과정입니다.

## 변경된 소스 내용

### 1. EventAdvice.java (신규)

**위치:** `bytebuddy-advice/src/main/java/com/javaagent/bytebuddy/advices/EventAdvice.java`

```java
package com.javaagent.bytebuddy.advices;

import com.javaagent.bytebuddy.helper.EventHelper;
import net.bytebuddy.asm.Advice;
import jakarta.servlet.http.HttpServletRequest;

/**
 * Spring Event Capture Advice
 *
 * DispatcherServlet.doService() 메서드를 계측하여
 * HTTP 요청의 헤더와 바디를 JSON 형식으로 출력합니다.
 */
public class EventAdvice {

    @Advice.OnMethodEnter(inline = true)
    public static void onMethodEnter(
            @Advice.Origin String method,
            @Advice.Argument(0) Object request
    ) {
        if (request instanceof HttpServletRequest) {
            HttpServletRequest httpRequest = (HttpServletRequest) request;

            try {
                EventHelper.printRequestInfo(httpRequest);
                EventHelper.printRequestHeaders(httpRequest);
                EventHelper.printRequestBody(httpRequest);
            } catch (Exception e) {
                System.err.println("{\"error\":\"" + e.getMessage() + "\"}");
            }
        }
    }
}
```

**변경사항:**
- 클래스명: `KernelAdvice` → `EventAdvice`
- Helper: `KernelHelper` → `EventHelper`
- 패키지: 동일 (`com.javaagent.bytebuddy.advices`)

### 2. EventHelper.java (신규)

**위치:** `bytebuddy-advice/src/main/java/com/javaagent/bytebuddy/helper/EventHelper.java`

```java
package com.javaagent.bytebuddy.helper;

import jakarta.servlet.http.HttpServletRequest;
import java.util.*;

public class EventHelper {

    public static void printRequestInfo(HttpServletRequest request) {
        System.err.println("{");
        System.err.println("  \"method\": \"" + request.getMethod() + "\",");
        System.err.println("  \"uri\": \"" + request.getRequestURI() + "\",");
        System.err.println("  \"queryString\": \"" +
            (request.getQueryString() != null ? request.getQueryString() : "") + "\",");
        System.err.print("  \"contentType\": \"" +
            (request.getContentType() != null ? request.getContentType() : "") + "\"");
        System.err.println();
        System.err.println("}");
        System.err.flush();
    }

    public static void printRequestHeaders(HttpServletRequest request) {
        System.err.println("{\"headers\":{");

        List<String> headerNames = Collections.list(request.getHeaderNames());
        if (headerNames.isEmpty()) {
            System.err.println("}");
            return;
        }

        for (int i = 0; i < headerNames.size(); i++) {
            String headerName = headerNames.get(i);
            List<String> headerValues = Collections.list(request.getHeaders(headerName));

            System.err.print("  \"" + headerName + "\":");
            if (headerValues.size() == 1) {
                System.err.print("\"" + escapeJson(headerValues.get(0)) + "\"");
            } else {
                System.err.print("[");
                for (int j = 0; j < headerValues.size(); j++) {
                    System.err.print("\"" + escapeJson(headerValues.get(j)) + "\"");
                    if (j < headerValues.size() - 1) System.err.print(", ");
                }
                System.err.print("]");
            }

            if (i < headerNames.size() - 1) {
                System.err.println(",");
            } else {
                System.err.println();
            }
        }

        System.err.println("}}");
        System.err.flush();
    }

    public static void printRequestBody(HttpServletRequest request) {
        System.err.print("{\"body\":\"");

        try {
            if (request.getContentLength() <= 0) {
                System.err.println("\"}");
                return;
            }

            // TODO: getInputStream() requires ServletInputStream class
            System.err.println("(body length: " +
                request.getContentLength() + " bytes - temporarily disabled)\"}");

        } catch (Exception e) {
            System.err.println("\"}");
        }

        System.err.flush();
    }

    private static String escapeJson(String s) {
        if (s == null) return "";
        return s.replace("\\", "\\\\")
                 .replace("\"", "\\\"")
                 .replace("\n", "\\n")
                 .replace("\r", "\\r")
                 .replace("\t", "\\t");
    }
}
```

**특징:**
- Jakarta Servlet API 사용 (Spring Boot 3.x 호환)
- 다중 헤더 값 지원 (배열 형태)
- JSON 이스케이프 처리
- `System.err` 사용 (Maven 로그에 출력)

### 문제 2: Agent 초기화 실패

**증상:**
```json
{
  "success": false,
  "message": "ERROR: Agent not initialized. Please attach to JVM first."
}
```

**원인 분석 과정:**

1. **Attach 성공 확인:**
```bash
curl -X POST http://localhost:9999/api/bytebuddy/attach \
  -d '{"pid": "9748"}'
# Response: {"success": true, "pid": "9748"}
```

2. **하지만 instrumentation이 null:**
```java
// ByteBuddyAgent.java
public static String createEventAdvice(String className, String methodName) {
    if (instrumentation == null) {
        return "ERROR: Agent not initialized.";
    }
    // ...
}
```

3. **로그 확인:**
```bash
tail -50 test-services/test1-service/test1.log
```

**로그에서 발견:**
```
[ByteBuddy] Agent initialized
[ByteBuddy] HTTP server started on port 10000
```

**핵심 발견:**
- `instrumentation`은 초기화됨 (`[ByteBuddy] Agent initialized`)
- 하지만 **agent-server(포트 9999)와 ByteBuddyAgent HTTP 서버(포트 10000)는 별개!**
- agent-server의 `createEventAdvice()`는 **자신의 클래스패스에 있는 ByteBuddyAgent**를 호출
- 실제 target JVM의 ByteBuddyAgent는 포트 10000에서 독립적으로 실행

**아키텍처 이해:**

```
┌─────────────────────────────────────────────────────────────┐
│                      agent-server (9999)                     │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  ByteBuddyController.createEventAdvice()              │   │
│  │    ↓                                                  │   │
│  │  ByteBuddyAgent.createEventAdvice() (static call)     │   │
│  │    ↓                                                  │   │
│  │  instrumentation == null! ❌                          │   │
│  │  (agent-server의 ByteBuddyAgent는 초기화되지 않음)     │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                  Target JVM (test1, PID 12280)              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  ByteBuddyAgent (agentmain으로 초기화됨)              │   │
│  │    - instrumentation ✅                              │   │
│  │    - HTTP Server on port 10000 ✅                    │   │
│  │    - EventAdvice 적용 가능 ✅                         │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**해결:**
```bash
# agent-server(9999) 대신 ByteBuddyAgent HTTP 서버(10000)에 직접 요청
curl -X POST http://localhost:10000/api/createEventAdvice \
  -H "Content-Type: application/json" \
  -d '{"className":"org.springframework.web.servlet.DispatcherServlet","methodName":"doService"}'

# Response: {"success": true, "message": "SUCCESS: Created event advice..."}
```

## 왜 이렇게 오래 시간이 걸렸는가?

### 1. 아키텍처 오해 (가장 큰 원인)

**잘못된 가정:**
- agent-server와 ByteBuddyAgent가 같은 프로세스에서 실행된다고 생각
- agent-server의 API가 target JVM의 ByteBuddyAgent를 직접 제어할 수 있다고 생각

**실제 아키텍처:**
```
1. agent-server (포트 9999)
   - Spring Boot REST API
   - attach 요청만 처리
   - 자신의 클래스패스에 있는 ByteBuddyAgent는 초기화되지 않음

2. ByteBuddyAgent HTTP 서버 (포트 10000)
   - target JVM 내에서 실행
   - agentmain으로 초기화됨
   - 독립적인 HTTP 서버로 계측 명령 처리
```

**시간 소요:** 약 1시간 (코드, 로그, 아키텍처 반복 분석)

---

### 2. 초기화 실패의 디버깅 어려움

**문제:**
- `instrumentation == null`인 이유를 바로 알 수 없음
- agent-server의 ByteBuddyAgent와 target JVM의 ByteBuddyAgent가 다르다는 것을 깨닫지 못함

**시도한 방법:**
1. 여러 번 재시작 → 실패
2. JAR 파일 재빌드 → 실패
3. 로그 확인 → 성공 (포트 10000 발견)

**시간 소요:** 약 30분

---

### 3. 컴파일 오류와 포트 충돌

**컴파일 오류:**
- DTO 중복 정의
- 찾는데 10분, 수정하는데 5분

**포트 충돌:**
- 여러 서비스가 동시에 실행
- 포트 확인하고 정리하는데 20분

---

### 4. 서비스 재시작 반복

**이유:**
- Java 프로세스가 모두 꼬여서 깔끔하게 재시작 필요
- 매번 포트 확인, 프로세스 종료, 재시작 반복

**해결:**
```bash
# 한 번에 모든 Java 프로세스 종료
taskkill //F //IM java.exe
```

**시간 소요:** 약 20분

### 2. 아키텍처 명확화

**agent-server (포트 9999):**
- 역할: JVM attach만 담당
- 엔드포인트: `POST /api/bytebuddy/attach`
- 제한: 계측 명령은 처리할 수 없음

**ByteBuddyAgent HTTP 서버 (포트 10000):**
- 역할: 실제 계측 명령 처리
- 엔드포인트:
  - `POST /api/createEventAdvice`
  - `POST /api/createSpan`
  - 등
- 위치: target JVM 내부

## 학습 포인트

### 1. Java Agent 아키텍처

```
JVM Attach
    ↓
VirtualMachine.attach(pid)
    ↓
vm.loadAgent(agentJar)
    ↓
agent-main() 호출 in target JVM
    ↓
ByteBuddyAgent.instrumentation 초기화
    ↓
독립 HTTP 서버 시작 (포트 10000)
```

### 2. 클래스패스와 클래스로더

- agent-server의 ByteBuddyAgent: agent-server의 ClassLoader로 로드
- target JVM의 ByteBuddyAgent: target JVM의 ClassLoader로 로드
- **두 개는 전혀 다른 인스턴스!**

### 3. 로그의 중요성

```
[ByteBuddy] HTTP server started on port 10000
```
이 한 줄이 모든 문제를 해결했다!

## 결론

1. **네이밍 변경 완료:** `KernelAdvice` → `EventAdvice`
2. **필터 계측 성공:** DispatcherServlet.doService() 계측
3. **아키텍처 이해:** agent-server와 ByteBuddyAgent의 분리 이해
4. **엔드포인트 발견:** 포트 10000의 ByteBuddyAgent HTTP 서버

**핵심 교훈:**
- Java Agent는 target JVM 내에서 독립적으로 실행됨
- 두 개의 HTTP 서버가 존재하며 각각 다른 역할을 함
- 로그를 자세히 확인하면 답이 있다!
