# 20260329 필터 계측 성공 - 요청 헤더/바디 JSON 출력

## 목표

test1 서비스가 받은 HTTP 요청의 헤더와 바디를 JSON 형식으로 출력하는 필터 계측 구현

## 최종 결과

```json
{
  "method": "GET",
  "uri": "/test1",
  "queryString": "userId=user123&data=testdata",
  "contentType": "application/json"
}
{"headers":{
  "host":"localhost:8081",
  "user-agent":"curl/8.18.0",
  "accept":"*/*",
  "content-type":"application/json",
  "x-custom-header":"custom-value",
  "authorization":"Bearer token123",
  "x-test-header":"test-value"
}}
{"body":""}
```

## 구현 개요

### 생성 파일

1. **KernelAdvice.java** (`bytebuddy-advice/src/.../advices/KernelAdvice.java`)
   - Spring Filter 계측을 위한 ByteBuddy Advice
   - `@Advice.OnMethodEnter`에서 요청 정보 캡처

2. **KernelHelper.java** (`bytebuddy-advice/src/.../helper/KernelHelper.java`)
   - `HttpServletRequest`에서 헤더/바디 추출
   - JSON 형식으로 출력

### 적용 대상

- **DispatcherServlet.doService** - 모든 HTTP 요청이 통과하는 Spring MVC 진입점

## 트러블슈팅 과정

### 문제 1: javax.servlet vs jakarta.servlet

**증상:**
```
package javax.servlet.http does not exist
```

**원인:** Spring Boot 3.x는 Java EE 대신 Jakarta EE를 사용합니다. `javax.servlet`이 아니라 `jakarta.servlet`을 사용해야 합니다.

**해결:**
```xml
<!-- bytebuddy-advice/pom.xml -->
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>
</dependency>
```

두 가지 모두 추가하여 Spring Boot 2.x/3.x 호환성 확보.

### 문제 2: Filter 클래스를 찾지 못함

**증상:**
```
[ByteBuddy] Total loaded classes: 9900
[ByteBuddy] jakarta.servlet.Filter is available
[ByteBuddy] Checked 9900 classes total
[ByteBuddy] Total filters found: 0
```

**원인:**
1. Spring Boot의 Filter들은 `jakarta.servlet.Filter`를 직접 구현하지 않습니다
2. Spring은 `FilterRegistrationBean`, `OncePerRequestFilter` 등을 통해 Filter를 래핑합니다
3. Agent attach 시점에는 Filter 클래스가 아직 로드되지 않았습니다

**시도한 방법:**
- `javax.servlet.Filter` 검색 → 실패
- `jakarta.servlet.Filter` 검색 → 실패
- `org.springframework.web.filter.*` 검색 → Filter 관련 클래스 발견하지만 실제 Filter 아님

**발견된 Filter 관련 클래스:**
```
org.springframework.web.filter.GenericFilterBean$FilterConfigPropertyValues
org.apache.catalina.core.ApplicationFilterConfig
org.apache.tomcat.websocket.server.WsFilter
jakarta.servlet.GenericFilter
```

이들은 실제 Filter가 아니라 Filter 설정/관련 헬퍼 클래스들입니다.

### 문제 3: SpanAdvice가 계속 적용됨

**증상:**
```
[ByteBuddy] *** ADVICE APPLIED TO org.springframework.web.servlet.DispatcherServlet.doService ***
>>> ADVICE ENTER: protected void org.springframework.web.servlet.DispatcherServlet.doService(...)
```

KernelAdvice를 적용하려고 했지만 SpanAdvice가 계속 적용되었습니다.

**원인:**
- `/api/createSpan` 엔드포인트가 SpanAdvice를 적용합니다
- HTTP 서버에 KernelAdvice를 적용하는 핸들러가 없었습니다

**해결:** HTTP 서버에 `/api/createKernelAdvice` 핸들러 추가

```java
server.createContext("/api/createKernelAdvice", exchange -> {
    String result = createKernelAdvice(className, methodName);
    // ...
});
```

### 문제 4: JSON 출력 형식 요구사항

**요구사항:**
- 이모티/문자열 대신 JSON 형식
- 헤더와 바디만 출력
- `System.err` 사용 (Maven 로그 출력)

**구현:**

```java
// KernelHelper.java
public static void printRequestHeaders(HttpServletRequest request) {
    System.err.println("{\"headers\":{");

    List<String> headerNames = Collections.list(request.getHeaderNames());
    for (int i = 0; i < headerNames.size(); i++) {
        String headerName = headerNames.get(i);
        String headerValue = request.getHeader(headerName);
        System.err.print("  \"" + headerName + "\":\"" + escapeJson(headerValue) + "\"");

        if (i < headerNames.size() - 1) {
            System.err.println(",");
        }
    }

    System.err.println("}}");
}

private static String escapeJson(String s) {
    return s.replace("\\", "\\\\")
             .replace("\"", "\\\"")
             .replace("\n", "\\n")
             .replace("\r", "\\r")
             .replace("\t", "\\t");
}
```

## 최종 해결 방법

### 1. Filter 대신 DispatcherServlet 계측

Filter를 찾지 못해 모든 HTTP 요청이 통과하는 **DispatcherServlet.doService**를 계측:

```bash
curl -X POST http://localhost:9999/api/createKernelAdvice \
  -H "Content-Type: application/json" \
  -d '{"className":"org.springframework.web.servlet.DispatcherServlet","methodName":"doService"}'
```

### 2. Jakarta Servlet 사용

Spring Boot 3.x 호환을 위해 `jakarta.servlet.http.HttpServletRequest` 사용:

```java
import jakarta.servlet.http.HttpServletRequest;

public class KernelAdvice {
    @Advice.OnMethodEnter(inline = true)
    public static void onMethodEnter(
            @Advice.Origin String method,
            @Advice.Argument(0) Object request
    ) {
        if (request instanceof HttpServletRequest) {
            HttpServletRequest httpRequest = (HttpServletRequest) request;
            KernelHelper.printRequestInfo(httpRequest);
            KernelHelper.printRequestHeaders(httpRequest);
            KernelHelper.printRequestBody(httpRequest);
        }
    }
}
```

### 3. JSON 형식 출력

`System.err.println`으로 JSON 출력 (Maven 로그에 표시됨):

```java
public static void printRequestInfo(HttpServletRequest request) {
    System.err.println("{");
    System.err.println("  \"method\": \"" + request.getMethod() + "\",");
    System.err.println("  \"uri\": \"" + request.getRequestURI() + "\",");
    System.err.println("  \"queryString\": \"" + request.getQueryString() + "\",");
    System.err.println("  \"contentType\": \"" + request.getContentType() + "\"");
    System.err.println("}");
}
```

## 주요 학습 포인트

### 1. Spring Boot 3.x 변경사항
- `javax.servlet.*` → `jakarta.servlet.*`
- Filter 구조가 복잡해져서 직접 구현 클래스를 찾기 어려움

### 2. ByteBuddy 계측 전략
- **Filter 계측이 어려운 경우** → 핵심 서블릿 계측
- `DispatcherServlet`은 모든 요청이 통과하는 이상적인 타겟

### 3. Agent JAR 배포
- `agent-server/` 디렉토리에 JAR 복사 필요
- Maven 빌드 후 복사 필수

### 4. 로그 출력
- Maven `spring-boot:run`은 `System.err`을 콘솔에 표시
- `System.out`은 로그 파일로만 감

## 추가 개선 가능 사항

1. **바디 읽기 문제:** InputStream은 한 번만 읽을 수 있어서 이미 읽은 경우 "(already read)"로 표시
2. **필터 체인:** 실제 Filter 클래스들을 찾아서 개별 계측 가능
3. **응답 출력:** HttpServletResponse도 계측하여 응답 헤더/바디 출력
4. **OpenTelemetry 연동:** 캡처한 정보를 Span 속성으로 저장
