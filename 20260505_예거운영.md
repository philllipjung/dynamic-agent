## 1. Jaeger MCP API 테스트

### 1.1 MCP 서버 접속 테스트

**초기화 응답**:
```json
{
  "serverInfo": {
    "name": "jaeger",
    "version": "v2.17.0"
  },
  "protocolVersion": "2024-11-05",
  "capabilities": {
    "logging": {},
    "tools": {
      "listChanged": true
    }
  }
}
```

✅ **결과**: MCP 서버 정상 작동

### 1.2 세션 초기화

**절차**:
1. `initialize` 메서드 호출 → Session ID 획득
2. `notifications/initialized` 전송
3. 이후 `tools/call`로 API 사용

**Session ID 예시**: `IHCXMDZXLBHBEDIXQCKQEUKRPR`

### 1.3 API 테스트 결과

| # | API | 상태 | 설명 |
|---|-----|------|------|
| 1 | health | ✅ 작동 | Server: jaeger, Status: ok, Version: v2.17.0 |
| 2 | get_services | ✅ 작동 | 3개 서비스 발견: batch-processor, jaeger, unknown_service:java |
| 3 | get_span_names | ✅ 작동 | service_name 파라미터 필요 (snake_case) |
| 4 | search_traces | ✅ 작동 | service_name 파라미터 필요, limit 파라미터 미지원 |
| 5 | get_trace_topology | ⚠️ 미테스트 | trace ID 필요 |
| 6 | get_trace_errors | ⚠️ 미테스트 | trace ID 필요 |
| 7 | get_span_details | ⚠️ 미테스트 | trace_id + span_ids[] 필요 |
| 8 | get_critical_path | ⚠️ 미테스트 | trace ID 필요 |

**성공률**: 3/8 완전 테스트 (API 호출 성공: 4/8)

### 1.4 중요 발견사항

#### 파라미터 네이밍 규칙
- **Jaeger MCP는 `snake_case` 사용** (Go 기반)
- ❌ 잘못된 예: `serviceName`, `spanId`, `traceId`
- ✅ 올바른 예: `service_name`, `span_id`, `trace_id`

#### 응답 포맷 (SSE - Server-Sent Events)
```
event: message
data: {"jsonrpc":"2.0","id":1,"result":{...}}
```

#### 파싱 방법
```bash
# sed로 data: 부분 추출
curl -s ... | sed 's/.*data: //'

# 또는 grep으로 필드 추출
curl -s ... | grep -o '"server":"[^"]*"' | cut -d'"' -f4
```

### 1.5 샘플 API 호출

#### Health Check
```bash
curl -s -X POST http://localhost:16687/mcp \
  -H "Content-Type: application/json" \
  -H "Mcp-Session-Id: $SESSION_ID" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"health","arguments":{}}}'
```

**응답**:
```json
{
  "structuredContent": {
    "server": "jaeger",
    "status": "ok",
    "version": "v2.17.0"
  }
}
```

#### Get Services
```bash
curl -s -X POST http://localhost:16687/mcp \
  -H "Content-Type: application/json" \
  -H "Mcp-Session-Id: $SESSION_ID" \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"get_services","arguments":{}}}'
```

**응답**:
```json
{
  "structuredContent": {
    "services": ["batch-processor", "jaeger", "unknown_service:java"]
  }
}
```

#### Get Span Names
```bash
curl -s -X POST http://localhost:16687/mcp \
  -H "Content-Type: application/json" \
  -H "Mcp-Session-Id: $SESSION_ID" \
  -d '{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"get_span_names","arguments":{"service_name":"batch-processor"}}}'
```

**응답**:
```json
{
  "structuredContent": {
    "span_names": [
      {"name": "POST", "span_kind": ""}
    ]
  }
}
```

### 1.6 한계점

#### 현재 데이터 부족
- 모든 서비스에서 trace 데이터가 없음
- `search_traces` 응답이 비어있음
- trace ID가 없어 topology, errors, critical_path 테스트 불가

#### 해결 방안
1. 샘플 애플리케이션 실행으로 trace 생성
2. microsim 또는 다른 trace generator 사용
3. 기존 trace가 있는지 OpenSearch 직접 조회
